# NKS Private Worker Outbound OpenVPN 원리 및 통신 흐름 정리

## 1. 목적

이번 과제의 본질은 아래입니다.

- `Private NKS Worker Node`가 직접 인터넷으로 나갈 수 없는 환경에서
- `Pod`의 외부 HTTP 요청을 `OpenVPN Server`를 통해 우회시켜
- 최종적으로 `Pod -> curl https://google.com`을 성공시키는 것

즉 단순히 `Worker Node`가 VPN에 붙는 것만으로 끝나는 과제가 아니라,

- `node egress`
- `Private URI image pull`
- `CoreDNS`
- `pod egress`

가 모두 이어져야 최종 성공입니다.

## 2. 전체 구성 개념

```text
Pod
  -> Worker Node
  -> tun0 (OpenVPN client)
  -> OpenVPN Server
  -> Internet

또는

Pod / kubelet / containerd
  -> Worker Node eth0
  -> Private DNS / NCR SGW / OBS SGW
```

핵심은 `Worker Node`가 두 가지 경로를 동시에 가져야 한다는 점입니다.

- 외부 인터넷 통신
  - `tun0`
- 내부/사설 서비스 통신
  - `eth0`

## 3. 왜 Worker Node 직접 OpenVPN client 방식인가

여러 방안 중 이번 과제에서 채택한 것은 `Worker Node 직접 OpenVPN client` 방식입니다.

이유:

- `Pod` 외부 통신 경로가 가장 단순함
- `node`, `kubelet`, `containerd`, `pod`가 같은 egress 정책을 공유함
- `Issuer API`와 연결하면 autoscale도 자동화 가능함
- 실제로 `Private URI image pull`과 `pod curl google.com`까지 검증 완료됨

## 4. Basic Auth -> node-token -> node-bundle 원리

현재 자동 발급 구조는 3단계입니다.

### 4.1 Basic Auth

이 단계의 목적은 인증서 발급이 아니라 **bootstrap 진입**입니다.

받는 파일:

- `bootstrap-root-ca.pem`
- `worker-egress-bootstrap.sh`
- `node-runtime-ubuntu2204-amd64.tar.gz`

즉 `Basic Auth`는 공통 파일을 안전하게 가져오기 위한 1차 관문입니다.

### 4.2 node-token

2차 스크립트가 node의 기본 정보를 모아 `Issuer API`에 전달합니다.

예:

- `node_id`
- `node_group`
- `cluster`
- `instance_id`
- `local_hostname`
- `private_ip`

서버는 이 값을 검증한 뒤 짧은 만료의 1회성 token을 발급합니다.

즉 `node-token`은 “이 node가 발급 대상인지 확인하는 단계”입니다.

### 4.3 node-bundle

2차 스크립트는 방금 받은 1회성 token으로 다시 `Issuer API`를 호출합니다.

서버는 여기서:

- token 유효성 확인
- 1회성 사용 여부 확인
- node 정보 재확인
- Easy-RSA로 node 전용 cert/key 발급

응답:

- `ca.crt`
- `client.crt`
- `client.key`
- `tls-crypt.key`
- `bundle-info.txt`

즉 `node-bundle`이 실제 OpenVPN 접속용 인증서를 받는 단계입니다.

## 5. worker node 부팅 시 실제 흐름

```text
node boot
  -> 1차 user script
  -> bootstrap endpoint에서 2차 스크립트 수신
  -> bootstrap CA 준비
  -> runtime package 확인/설치
  -> node-token 요청
  -> 1회성 token 수신
  -> node-bundle 요청
  -> cert/key bundle 수신
  -> /etc/openvpn/client/pki 배치
  -> worker-egress.conf 생성
  -> openvpn-client@worker-egress 기동
  -> split DNS 적용
```

## 6. Pod에서 외부 URL 호출 시 통신 흐름

이번 과제의 핵심 성공 흐름은 아래입니다.

```text
Pod
  -> CoreDNS에 google.com 질의
  -> node split DNS 기준 public name은 tun0 DNS 사용
  -> Pod outbound는 worker node에서 tun0 경유
  -> OpenVPN tunnel
  -> OpenVPN Server
  -> SNAT / Internet
  -> google.com 응답
```

중요:

- Pod는 직접 `OpenVPN Server`와 통신하지 않습니다.
- 실제 VPN 종단은 `Worker Node`입니다.
- Pod의 외부 트래픽이 node 라우팅을 따라 `tun0`로 나가는 구조입니다.

## 7. 왜 split DNS가 필요한가

### 7.1 일반 원리

일반적으로 `OpenVPN client`가 붙고 기본 외부 경로가 `tun0`이면,

- `node curl https://www.google.com`

은 되는 게 맞습니다.

즉 모든 node 설치형 OpenVPN 방안이 `split DNS`를 반드시 요구하는 것은 아닙니다.

### 7.2 이번 과제의 추가 요구

이번 과제는 `Pod curl google.com`만이 아니라, 그 전에 아래도 충족해야 했습니다.

- `Private NKS`
- `Private URI image pull`
- `NCR / OBS`

이미지 pull은 `Pod`가 아니라 `node의 kubelet / containerd`가 합니다.  
따라서 node가 아래 이름도 정상 처리해야 합니다.

- `private-<REGISTRY_ID>...container.nhncloud.com`
- `kr1-api-object-storage.nhncloudservice.com`

이 이름들을 public DNS로 보내면 안 되므로, 현재 표준은 아래처럼 나눕니다.

- `eth0 -> Private DNS`
  - `openstacklocal`
  - `container.nhncloud.com`
  - `nhncloudservice.com`
- `tun0 -> public DNS`
  - 일반 public domain

즉 `split DNS`는 이번 표준 경로에서 **Private URI image pull과 public egress를 동시에 만족시키기 위해** 필요합니다.

## 8. CoreDNS / cluster.local / Ingress host 원칙

### 8.1 cluster.local

`kubernetes.default.svc.cluster.local` 같은 이름은 `CoreDNS`가 클러스터 내부에서 직접 처리합니다.

즉 현재 `split DNS`는:

- `cluster.local` 자체를 대체하는 것이 아니라
- 외부 도메인 또는 private/public upstream 선택에만 영향

을 줍니다.

### 8.2 ingress host

`app.example.internal` 같은 `Ingress host`는 `CoreDNS`가 자동 생성하지 않습니다.

따라서 별도로:

- `Private DNS`
- 또는 공인 DNS

레코드가 있어야 합니다.

## 9. 왜 Pod 생성 전 image pull이 먼저 중요했는가

이번 과제에서는 `Pod`를 띄우기 전 단계에서 이미 node가 다음을 만족해야 했습니다.

1. `private registry` 이름을 올바르게 풀 것
2. `OBS SGW`로 접근 가능할 것
3. image pull이 성공할 것

이게 실패하면 test pod 자체가 만들어지지 않으므로, `Pod curl google.com`은 그 다음 문제입니다.

즉 실제 검증 순서는 아래가 맞습니다.

1. `node`가 VPN에 붙는지
2. `node`에서 `google.com`이 되는지
3. `Private URI image pull`이 되는지
4. test pod가 생성되는지
5. pod에서 `cluster.local`이 되는지
6. pod에서 `google.com`이 되는지
7. pod에서 `curl https://www.google.com`이 되는지

## 10. reboot과 scale-out에서의 동작

### 10.1 같은 node reboot

- 기존에 발급된 cert 파일을 그대로 재사용
- bootstrap token을 다시 받을 필요 없음
- 보통 기존 OpenVPN client 설정으로 재연결

### 10.2 새 node scale-out

- 새 node는 새 metadata를 가짐
- 따라서 `node-token -> node-bundle`을 다시 수행
- 새 node 전용 cert가 발급됨

### 10.3 동시 scale-out

현재 `Issuer API` 구현은 요청-응답형 동기 구조입니다.

- `node-token`은 비교적 빠르게 여러 개 처리 가능
- 실제 `node-bundle` 발급은 CA 서버에서 직렬화될 수 있음

따라서 여러 node가 동시에 늘어나면 VPN 연결 완료 시점은 순차적으로 밀릴 수 있습니다.

## 11. 현재 구조의 의미

이번 과제에서 검증된 구조는 아래 의미를 가집니다.

- 폐쇄망 성격의 `Private NKS Worker`에서도 외부 API 통신 가능
- `OpenVPN Server`를 인터넷 출구로 사용하는 우회 구조 검증 완료
- `Issuer API` 기반으로 autoscale worker 인증서 자동 발급 가능
- `Private URI image pull`과 public outbound를 같은 node에서 병행 가능
- 최종적으로 `Pod curl https://google.com` 성공

즉 이번 과제는 단순한 VPN 연결 검증이 아니라,

- `node lifecycle`
- `PKI 자동 발급`
- `node routing`
- `split DNS`
- `CoreDNS`
- `image pull`
- `pod egress`

가 모두 이어진 end-to-end 검증이라고 볼 수 있습니다.
