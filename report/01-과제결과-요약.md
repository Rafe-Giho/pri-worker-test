# NKS Private Worker Outbound OpenVPN 과제 결과 보고

## 1. 과제 개요

- 과제 목표
  - `NHN Testbed`에서 `NKS`를 구성하고 `Worker Node`는 `PrivZone`에 배치
  - `Pod`에서 `curl https://google.com` 응답 확인
- 제약 조건
  - `Worker Node`가 직접 인터넷으로 나가지 않는 구간을 전제로 구성
  - 외부 통신은 `OpenVPN Server`를 통한 우회 경로로 구성

## 2. 최종 구현 방안

최종 구현 방안은 아래와 같습니다.

- `주 방안`: `Worker Node`에 `OpenVPN client`를 직접 설치
- `자동화 방식`: `NKS user script + bootstrap 2차 스크립트`
- `인증서 발급 방식`: `Issuer API` 기반 full-auto
- `인증 흐름`: `Basic Auth -> node-token -> node-bundle`
- `DNS 방식`: `split DNS`
  - `eth0 -> Private DNS`
  - `tun0 -> public DNS`

즉 현재 표준 경로는 아래와 같습니다.

```text
Private NKS Worker Node
  -> bootstrap endpoint에서 공통 파일 수신
  -> Issuer API에서 node별 OpenVPN bundle 자동 발급
  -> worker-egress OpenVPN client 기동
  -> pod 외부 통신은 tun0 -> OpenVPN Server -> Internet
```

## 3. 구성 요약

### 3.1 공통 구성

- `Public VPC OpenVPN Server`
  - 역할: VPN 종단, 복호화, NAT, 인터넷 출구
- `Private VPC CA / Bootstrap / Issuer API`
  - 역할: PKI, bootstrap 파일 배포, worker cert 자동 발급
- `Private DNS`
  - 역할: `NCR / OBS / 내부 도메인`을 `eth0` 경로로 유지
- `NCR / OBS Service Gateway`
  - 역할: `Private URI image pull`

### 3.2 검증 완료된 실환경 예시

- 클러스터명: `ta-sgh-pri-cls`
- 주 방안 node group: `default-worker`
- `CA / Bootstrap / Issuer API`: `172.16.200.44`
- `Issuer API`: `https://172.16.200.44:8443`
- `bootstrap endpoint`: `https://172.16.200.44/ovpn`
- `OpenVPN Server private IP`: `192.168.200.25`
- `Private DNS server`: `172.16.0.105`

주의:

- 보고서에는 민감정보를 넣지 않았습니다.
- 비밀번호, Access Key, CA passphrase는 운영 문서/보안 저장소에서 별도 관리해야 합니다.

## 6. 왜 이 방안을 채택했는가

이번 과제에서는 여러 구현 방향 중 아래 3가지를 비교했습니다.

- `A`: `Worker Node`에 `OpenVPN client` 직접 설치
- `B`: `Gateway VM`을 next hop으로 두는 방식
- `C`: `Pod sidecar`에 `OpenVPN client`를 두는 방식

최종적으로 `A`를 채택한 이유:

- 과제 목표가 `Pod 외부 URL 호출 성공`까지 빠르게 검증하는 것이었음
- `A`는 경로가 가장 단순하고 `node/pod` 모두 같은 egress 정책으로 검증 가능함
- `Issuer API`와 결합하면 autoscale에도 대응 가능함
- 실제로 `Private URI image pull + pod curl google.com`까지 end-to-end 검증 완료됨

## 7. 핵심 원리 요약

### 7.1 OpenVPN 자체

- `Worker Node`가 `OpenVPN client`
- `OpenVPN Server`가 인터넷 출구
- node의 기본 외부 egress는 `tun0`로 우회

### 7.2 split DNS

- 일반적으로 `tun0` default route만 잡혀도 `node curl google.com`은 가능
- 하지만 이번 과제는 `Private URI image pull`도 함께 만족해야 함
- 따라서 `NCR / OBS / 내부 도메인`은 `eth0`, public domain은 `tun0`으로 나누는 `split DNS`가 필요

### 7.3 자동 발급

- bootstrap 단계에서는 `Basic Auth`로 공통 파일 수신
- 인증서 발급 단계에서는 `node-token -> node-bundle` 2단계
- 새 worker node가 생겨도 동일 흐름으로 node 전용 cert를 자동 발급받음

## 8. 현재 구조의 장점과 한계

### 8.1 장점

- `Private NKS` 환경에서 외부 API 통신 가능
- `Private URI image pull`과 public outbound를 같은 node에서 동시에 처리 가능
- 새 worker node scale-out 시 인증서 자동 발급 가능
- 기존 node reboot 시 기존 cert 재사용 가능

### 8.2 한계

- 현재 `Issuer API`는 요청-응답형 동기 처리
- 동시에 여러 node가 scale-out되면 실제 `node-bundle` 발급은 순차적으로 밀릴 수 있음
- `Gateway VM`, `Pod sidecar`는 설계/템플릿은 정리됐지만 실구현 검증은 아직 worker만 완료
